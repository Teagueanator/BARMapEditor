## ADR-025 — Starter texture pack

**Status:** Proposed (research) — 2026-05-18

**Context:** F4 (splat painting) ships in Phase 3 of the editor roadmap and is unusable without a populated tile palette; we need a "first-launch produces a believable map" path before user-import (F23) lands in Phase 6. Research-question B confirms that BAR maps universally use the engine's DNTS path (`splatDetailNormalTex1..4`, per https://springrts.com/wiki/Mapdev:splatdetailnormals — "`splatDetailTex` will be unused if any of `splatDetailNormalTex[1-4]` are present") with tile filenames sourced from the community Nobiax/Beherith DNTS pack (https://springrts.com/phpbb/viewtopic.php?t=34264). The BAR mapping guide normatively states: "Most DNTS textures are 512 x 512 or 1024 x 1024 px and in DDS, PNG or TGA format" (https://www.beyondallreason.info/guide/mapping-1-file-structure-prerequisites). A bundled starter pack lets a new user place real terrain without sourcing the Nobiax pack, learning the DNTS workflow, or reading the Beherith guide first.

**Alternatives:**
- **Bundled CC0 pack (chosen)** — predictable disk footprint, zero attribution UX, redistributable in AppImage/installer.
- **In-app downloader** — defers disk cost but breaks offline first-launch and creates network-failure UX paths; rejected for the starter experience but retained for Phase 6 (F23 user-import) which is a polish path.
- **User-import-only** — turns first-launch into a scavenger hunt; rejected.
- **Fork the Nobiax/Beherith DNTS pack** — CC0 and proven in-engine, but the pack is delivered as pre-baked DNTS DDS (normal-in-RGB, diffuse-in-alpha) which the editor cannot edit, recompose, or preview as diffuse without an unbake step. We instead bundle CC0 diffuse sources and let the pipeline produce DNTS at compile.

**Consequence:**
- 16 textures bundled via `scripts/fetch-textures.sh` into `tools/textures/`. Sources are ambientCG 1K JPG/PNG ZIPs; the script extracts only the `*_Color.*` (diffuse) member from each ZIP and discards the bundled normal/roughness/AO/displacement maps (the editor synthesises DNTS-format outputs from the diffuse via the existing barme-pipeline at compile time). Total extracted disk footprint ≈ **22 MB**, well under the 50 MB budget. Licence audit table below — all entries CC0-1.0, no per-texture attribution UX needed (ambientCG's per-site licence policy is uniform: "All ambientCG assets are provided under the Creative Commons CC0 1.0 Universal License", https://docs.ambientcg.com/license/).
- Splat brush UI defaults to this palette. Since the Recoil engine caps splat detail at **4 channels per map** (RGBA of the distribution texture — confirmed https://springrts.com/wiki/Mapdev:splatdetailnormals and https://en.wikipedia.org/wiki/Texture_splatting), the 16-tile palette is presented as four biome groups of four; the user picks up to 4 active tiles per map. User-import (F23 / Phase 6) is the polish path for arbitrary slots.
- **PyMapConv plumbing implications:** PyMapConv consumes the *full-map* texture (`-t intex`, "Sides must be multiple of 1024 long", https://springrts.com/phpbb/viewtopic.php?t=21458) and is **not** the splat-tile compiler. Splat detail tiles ship inside the `.sd7` and are referenced from `mapinfo.lua` `resources.splatDetailNormalTex` directly — PyMapConv only writes the SMF/SMT. Therefore the editor's pipeline (not PyMapConv) is responsible for converting the bundled JPG/PNG diffuse to DNTS-format DDS (normal-in-RGB, high-pass-filtered diffuse-in-alpha, per Beherith's verbatim spec: "The diffuse is highpass filtered to reduce tiling when zoomed out", https://springrts.com/phpbb/viewtopic.php?t=34264) and depositing them in the `.sd7` `maps/` directory alongside the SMF/SMT.
- Source diffuse files are stored at 1024² (the BAR-normative tile size) in **sRGB** colour space; the pipeline linearises during DXT/BC1 compression via Compressonator. No BMP intermediates are required for splat tiles (PyMapConv's BMP convention applies to the full-map intex/heightmap/metalmap pipeline, not to splat tiles loaded by the engine at runtime).

### Excluded but considered

- **Nobiax / Beherith DNTS pack** (https://springrts.com/phpbb/viewtopic.php?t=34264, "license PD"; underlying Nobiax pack independently confirmed CC0 at https://springrts.com/phpbb/viewtopic.php?t=33492). Native to BAR and would "just work" via copy-into-`.sd7`, but ships as pre-baked DNTS DDS (normal-in-RGB, diffuse-in-alpha) which the editor cannot preview as diffuse without an unbake step; bundling raw CC0 diffuse sources is strictly more flexible and yields the same DNTS output via the editor pipeline.
- **Poly Haven outdoor library** (https://polyhaven.com/license, CC0). Excellent photogrammetry quality, but per the Poly Haven asset pages the 1K ZIPs for the relevant biome assets measure 77.67 MB (aerial_grass_rock), 81.84 MB (forrest_ground_01), 88.17 MB (forest_ground_04) and 99.59 MB (rocky_terrain_02) — a single asset would blow the 50 MB budget on its own, because Poly Haven ships both GL and DX normal pairs plus 16-bit displacement.
- **OpenGameArt "Seamless Terrain" sets** — mixed CC-BY / CC-BY-SA licences requiring per-asset attribution; disqualified by the no-attribution-UX constraint and the share-alike taint risk on user `.sd7` outputs.

### Open questions for implementation

1. **DNTS bake pipeline ownership.** The editor's barme-pipeline must produce DNTS-shape DDS from the bundled diffuse sources: derive a normal map (Sobel on luminance is the obvious default) and high-pass-filter the diffuse into the alpha channel. Compressonator BC3/DXT5 is the target. Verify the high-pass cutoff empirically — Beherith's verbatim instruction is only "The diffuse is highpass filtered to reduce tiling when zoomed out" (https://springrts.com/phpbb/viewtopic.php?t=34264); no radius/sigma value appears anywhere in the wiki or the release post, confirming this remains an implementation-tuning question.
2. **Verify ambientCG asset IDs at CI time.** `scripts/fetch-textures.sh` should HEAD-check each `_1K-JPG.zip` / `_1K-PNG.zip` URL and confirm each ZIP contains a `*_Color.*` member at CI time; ambientCG occasionally renumbers child assets and reprocesses color maps (e.g. Ground037's color map "was re-processed for better tiling on Jan 24th 2021", https://ambientcg.com/view?id=Ground037).
3. **Default texScales / texMults.** The Recoil engine default is `{0.02, 0.02, 0.02, 0.02}` — confirmed in two independent primary sources: the engine wiki (https://springrts.com/wiki/Mapdev:mapinfo.lua) and Beherith's original DNTS announcement which states verbatim "SplatTexScales defaults to (0.02 0.02 0.02 0.02), this is the current dtex resolution. Lower values mean larger size. Adjustable per channel" (https://springrts.com/phpbb/viewtopic.php?t=22564). Published BAR maps use values down to `0.0015` per channel — the lhog/spring-map-pbr "Enceladus" mapinfo.lua contains exactly `texScales = {0.004, 0.007, 0.008, 0.0015}` with `texMults = {0.18, 0.36, 0.36, 0.026}`. The editor should expose per-slot sliders defaulting to 0.02 with a hint that BAR authors typically use far smaller values.
4. **splatDistrTex authoring.** Out of scope for ADR-025 (covered by F4 splat-painting), but note: the engine reads RGBA channels of `splatDistrTex` as four greyscale masks routed to slots 1–4 in declaration order ("The RGBA Channels are individual greyscale distribution maps which define which detail textures to splat onto the ground", https://springrts.com/wiki/Mapdev:splatdetail). The editor must enforce this mapping at export.
5. **Specular texture requirement.** The legacy `splatDetailTex` path requires a `specularTex` to be defined ("Splat Detail textures requires a specular texture Mapdev:specular to exist and be defined in order to be enabled", https://springrts.com/wiki/Mapdev:splatdetail). The DNTS path appears not to require it, but verify against the current Recoil renderer (rts/Map/SMF/SMFRenderState.cpp in https://github.com/beyond-all-reason/RecoilEngine) before shipping — flagged as implementation-phase test.
6. **Hypothesis to verify:** Slot 14 ("ground-alien-orange") currently maps to ambientCG Ground003 as a placeholder. For BAR's signature alien-orange aesthetic (Gecko Isle Remake, Throne, Quicksilver) we may want to either ship an additional CC0 asset specifically authored for alien terrain or have the pipeline HSL-shift Ground003 toward orange at install. Decide in Phase 4 once the splat-preview renderer is in place to A/B the two approaches in-engine.

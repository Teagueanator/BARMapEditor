# **UX Recommendations for BAR Map Editor (Phase 3\)**

## **1\. Layout — top-level window structure**

The current interface architecture of the BAR Map Editor suffers from severe context collapse, heavily violating the spatial memory expectations of a modern Digital Content Creation (DCC) application. By stacking Project metadata, Heightmap validation, Render properties, Tool selection, Sculpt brush settings, Start position hierarchies, Procedural generation logic, and Build actions into a single scrolling 280-pixel vertical side panel, the application forces the user to continuously scroll and visually parse unrelated data streams. The user's primary mental model—manipulating a 3D terrain environment—is constantly interrupted by the mechanical friction of hunting for buried UI elements.  
To resolve this cognitive overload, the interface must be fundamentally restructured into an industry-standard layout. This approach is universally validated by established terrain and 3D authoring tools such as Blender 1, Unity Terrain Tools 2, and Houdini.3 Unlike competing web-based projects such as the Vue/WebGL-driven bar-editor fork 4, which can rely on flexible HTML DOM reflowing, this desktop application must strictly adhere to the deterministic capabilities of the egui immediate-mode framework. The redesign must entirely avoid unstable third-party docking solutions in favor of robust, natively supported geometric panels.  
The prescribed top-level window structure utilizes the following strict egui panel hierarchy, evaluated in this exact order during the application loop to ensure deterministic rendering and UI scaling:

1. **Top Bar (egui::TopBottomPanel::top)**: Reserved for global application state, primary menus, and high-level execution commands (Build/Preview).  
2. **Bottom Strip (egui::TopBottomPanel::bottom)**: Reserved for localized status feedback, contextual tool hints, and raw camera coordinates.  
3. **Global Toolbar (egui::SidePanel::left \- fixed width)**: A narrow vertical strip containing primary tool selection icons (e.g., Sculpt, Place, Paint, Select).  
4. **Tool Properties (egui::SidePanel::left \- resizable)**: A context-dependent panel positioned immediately adjacent to the Global Toolbar that displays only the specific parameters for the currently active tool.  
5. **Inspector (egui::SidePanel::right \- resizable)**: A persistent panel detailing global map properties, structural hierarchies (like start positions), and procedural generation settings.  
6. **Viewport (egui::CentralPanel)**: The 3D wgpu-rendered canvas occupying the remainder of the central window space.

The Top Bar replaces the simplistic, legacy menu strip. It requires structural division into three horizontal regions utilizing the ui.horizontal() layout mechanism. The left region houses standard dropdown menus. The center region introduces Global Mode Toggles, most notably the Symmetry configuration. Placing symmetry in the absolute center of the top bar elevates it to a global map state, communicating its profound effect on all subsequent user actions. The right region isolates the Build, Install, and Preview execution block. Elevating this block out of the scrolling side panel guarantees it is perpetually visible, actively communicating the map's compilation status and resolving the user frustration of "buried" build workflows.  
The newly introduced Bottom Strip serves as the primary discoverability engine for novice map designers. It features contextual text that updates dynamically based on the active tool and keyboard modifier states. If the Sculpt tool is active, the left side of the bottom strip explicitly broadcasts the interaction model. The right side of the strip displays the current camera telemetry and the raw world-space coordinates of the cursor hover point, excising this granular data from the primary Sculpt settings where it previously cluttered the interface.  
The Inspector panel on the right margin houses persistent map data that must remain accessible regardless of the active tool state. This mirrors the Unity Inspector paradigm 2, where global scene data—such as project metadata, the start position team tree, and the procedural generation evaluator—resides on the right. This structural choice keeps the left side of the application focused purely on immediate brush and mechanical tool adjustments, creating a clean cognitive separation between *how* the user interacts (tools on the left) and *what* the user is building (data on the right).

## **2\. Tool-mode model — Sculpt vs Place vs (future) Paint vs Select**

The application's current interaction model is restricted by a flat hierarchy, utilizing a simple radio button to toggle between sculpting and start-position placement. As the editor matures through Phase 3 (Splat mapping) and Phase 4 (Metal, Geometry, and Feature placement), retaining this radio-button design will result in unmanageable interface bloat. To preempt this structural failure, the editor requires a robust, state-driven tool-mode model represented by a vertical toolbar on the far-left edge of the screen, heavily adapting the standard interaction paradigms of raster editors like Krita 5 and the tool-shelf layout of Blender.1  
The underlying software architecture must represent the active tool as an exclusive state machine. Only one primary tool context can be active at any given frame, guaranteeing that canvas interactions are entirely deterministic and observable. The interface must dynamically hide and reveal the Tool Properties panel contents based on a pattern match against the active tool state enum.  
The vertical Global Toolbar must be designed to accommodate the tool ecosystem scaling gracefully through the application's maturity. The layout at Phase 5 maturity dictates the following structural mapping:

| Tool Designation | Interface Icon Concept | Primary Viewport Action | Secondary Viewport Action | Development Phase |
| :---- | :---- | :---- | :---- | :---- |
| **Sculpt** | Mountain Profile | Displace terrain height geometry | Smooth terrain height variance | Phase 1 (MVP) |
| **Level** | Flat Brush / Trowel | Flatten terrain to absolute target | Sample relative height under cursor | Phase 1 (MVP) |
| **Ramp** | Sloped Line | Drag vector to create inclined ramp | Snap ramp vector to discrete grid | Phase 3 |
| **Place** | Target Reticle | Drop start position marker | Move existing position marker | Phase 1 (MVP) |
| **Texture** | Paint Roller | Paint splatmap texture layer | Erase splatmap texture layer | Phase 3 |
| **Metal** | Hexagonal Node | Paint metal distribution layout | Erase metallic distribution | Phase 4 |
| **Feature** | Tree / Object | Place individual rendering feature | Paint localized feature cluster | Phase 4 |
| **Select** | Marquee Bounding Box | Box-select placed entities | Add multiple entities to selection | Phase 4 |

Managing the transitions between these distinct tool states requires precise handling of canvas rendering to prevent user frustration. A primary complaint indicates that switching to the Sculpt tool leaves start-position markers visible but unclickable, leading to confusion regarding application state. This represents a classic violation of UI affordance; if an object's visual representation remains identical, the user inherently assumes its interactive properties remain identical.  
The prescribed architectural fix requires binding the rendering state of viewport overlays directly to the active tool state. When a structural tool like "Place" or "Select" is active, start positions, features, and metal spots must render at full opacity, complete with hover-state highlighting and outline strokes. Raycasting from the cursor must prioritize intersecting these entities. Conversely, when a geometric or textural tool like "Sculpt" or "Texture" is active, the heightmap becomes the exclusive raycast target. To visually communicate this state change, all entity markers must immediately transition to a "ghosted" rendering state. By rendering these markers at reduced opacity, transitioning them to monochrome, and disabling their hover-state responses, the editor implicitly communicates that they are locked and uneditable. This resolves the behavioral confusion without entirely removing the structural context of the map from the user's view.

## **3\. Symmetry as a global mode**

Within the context of real-time strategy map design, symmetry is not a localized brush setting; it is the foundational architectural constraint governing the entire map topology. The Beyond All Reason community almost exclusively constructs maps engineered for strict competitive fairness, requiring perfect mirroring of terrain, texturing, metal layout, and starting positions.6 The current editor architecture buries symmetry configuration within the Sculpt section, inducing profound user errors as designers fail to realize that symmetry implicitly governs non-sculpting actions like start-position placement.  
Historical analysis of similar domain tools reveals the danger of localized symmetry. In the Planetary Annihilation system designer, UI obfuscation regarding multiple axes of symmetry caused significant friction for map creators attempting to build balanced worlds.7 To rectify this, the BAR Map Editor must elevate Symmetry to a hyper-visible, global application state that universally applies to every spatial mutation executed by the user.  
Symmetry settings must be completely excised from the specific Tool Properties panels. Instead, they must be implemented as a persistent, highly visible control block located in the exact center of the Top Bar. This block consists of a primary toggle switch enabling or disabling global symmetry, followed by a dropdown menu exposing the requisite operational modes (None, Horizontal, Vertical, Quad, Diagonal, Diagonal Rotational, and Rotational). When a rotational mode is selected, an egui::DragValue slider dynamically appears to allow integer configuration of the rotational fold count, mirroring the parameterization found in Blender's advanced sculpting toolset.8  
Elevating the interface controls is insufficient without rigorous on-canvas visualization. Without persistent visual feedback projected onto the 3D viewport, users cannot accurately gauge the center line or exact mirrored coordinates of their geometric strokes. The implementation team must leverage the wgpu rendering backend to draw persistent, depth-aware guide lines overlaying the heightmap geometry.  
For mirroring operations, the editor must draw a crisp, dashed axis line traversing the entire map along the mathematical line of symmetry. This line requires a custom shader overlay that disables depth testing (depth\_write\_enabled: false, depth\_compare: wgpu::CompareFunction::Always) ensuring it is never visually obscured by intersecting terrain peaks. For rotational symmetry, a prominent reticle must be rendered at the exact coordinate of rotation, with faint radial lines extending outward corresponding to the specified rotational fold count.  
Crucially, the editor must project synchronized "ghost" cursors at every mathematically symmetric point during active tool hover. If Quad symmetry is engaged, the user must see four independent brush outlines translating simultaneously across the terrain surface. This provides absolute, real-time visual confirmation of the spatial symmetry calculation before the user commits a destructive stroke to the heightmap.9

## **4\. Start-position editor redesign**

The current Start Position implementation represents the most critically inadequate feature relative to the specific requirements of the Beyond All Reason mapping ecosystem. The existing data model erroneously assumes a one-to-one mapping between a team and a symmetry image. Real-world competitive maps demand complex configurations, predominantly relying on 8v8 team structures resulting in sixteen discrete coordinate spots across two distinct sides, or Free-For-All configurations requiring multiple symmetric distributions.6  
The underlying data schema and the user interface must be completely rebuilt to accurately reflect the AllyTeam to Team to StartSpot hierarchical dependency inherent to the Recoil engine. The Right Inspector panel must house an interactive tree-view representing this structural map data. The AllyTeam represents an overarching faction or alliance side, mapping to a specific global identifier and team color. The Team represents an individual player slot within that AllyTeam, mapping to a specific precise start coordinate on the three-dimensional heightmap.  
The interface must facilitate rapid generation of these hierarchies. A Configuration Preset dropdown located at the top of the Inspector should offer standard architectural templates, such as 1v1, 8v8, and FFA configurations. Selecting the 8v8 preset instantly populates the Inspector's tree-view with two AllyTeam blocks, each containing eight pre-configured Team slots awaiting coordinate assignment.  
Manually placing sixteen individual coordinates is tedious and introduces a high probability of microscopic asymmetric drift. The Place tool must be fundamentally upgraded to leverage the newly established Global Symmetry mode combined with advanced vector placement interactions. The tool must support placing multiple distinct Team markers within the identical symmetry sector, automatically assigning these markers sequentially to the active AllyTeam tree.  
The system must automatically enforce symmetric multiplication. If a designer is constructing an 8v8 map with point symmetry, they configure the eight coordinates for AllyTeam 1 within one quadrant of the map. The editor's operational logic simultaneously generates and locks the opposing eight coordinates for AllyTeam 2 in the mirrored quadrant, mathematically guaranteeing perfect competitive spacing. Furthermore, drawing upon the region-selection and spatial array concepts seen in voxel editors like WorldEdit, the tool must support drag-painting. The user clicks and drags a vector line across the terrain; the editor evaluates the length of the vector and distributes the required number of unplaced start positions equidistantly along that spatial line.  
A seamless, bi-directional visual feedback loop between the egui Inspector UI and the wgpu viewport is mandatory to prevent spatial disorientation. When the user hovers their cursor over a specific Team entry in the textual Inspector, the corresponding 3D geometric marker on the canvas must visually pulse—scaling up by a percentage and increasing its shader emission intensity—to identify its location in physical space. Conversely, hovering over a 3D marker on the canvas must trigger a callback that automatically scrolls the Inspector panel to the corresponding Team entry and lightly highlights its background plate. This ensures the user never loses the correlation between abstract list data and physical map geometry.

## **5\. On-canvas feedback**

A prominent user complaint regarding the Sculpt tool identifies the total absence of on-canvas brush size previewing. Operating a localized spatial transformation without a visual representation of the tool's area of effect reduces sculpting to blind guesswork. The interface must provide high-fidelity, real-time geometric overlays directly onto the 3D viewport, standardizing the application with professional environments like World Machine 11 and Krita.5  
Mapping two-dimensional egui elements perfectly onto three-dimensional, variable-elevation topography presents a technical challenge within the current stack, yet the UX requirement remains non-negotiable. The implementation architecture dictates a Deferred Decal Projection utilizing the wgpu backend as the primary solution. By passing the 3D cursor coordinate, the brush radius parameter, and the brush falloff curve into a custom decal shader, the engine can render a textured geometric ring that strictly conforms to the underlying elevation changes of the heightmap. This methodology provides a topologically accurate preview identical to the brush targeting systems utilized in Unity Terrain Tools.12  
Should decal projection prove too computationally expensive for the MVP performance budget, a fallback mechanism utilizing egui::PaintCallbackInfo is acceptable.13 This approach calculates the screen-space bounding box of the physical brush radius and leverages the egui::Painter to draw a two-dimensional egui::Shape::circle\_stroke overlay. While this graphic does not deform to the 3D terrain shape, it guarantees a pixel-perfect absolute radius indicator layered above the rendering context.  
The visual design of the brush cursor must adapt the precision paradigms established by Krita's cursor configuration settings.5 The BAR Map Editor cursor requires three distinct structural components. The Precision Reticle serves as a single-pixel exact center point indicating the precise raycast intersection with the terrain collision mesh. The Outer Boundary, or Effective Outline, acts as a solid geometric circle representing the absolute maximum radius of the brush operation. Finally, the Inner Falloff visualizes a secondary, dashed concentric circle representing the core strength of the brush before mathematical falloff attenuation begins. If the user manipulates the brush falloff profile to be sharper, this inner circle dynamically expands closer to the outer boundary.  
When the user actively scales the brush size through numerical dragging in the properties panel or via keyboard shortcut bindings, the on-canvas outline must scale in real-time. This immediate interpolation provides absolute visual confirmation of the mathematical parameter adjustment, closing the execution gap between setting a value and understanding its physical consequence on the terrain.

## **6\. Discoverability of camera controls**

Camera navigation controls in three-dimensional environments represent the highest initial cognitive barrier to entry. If a user cannot comfortably orient themselves within the spatial coordinate system, they cannot effectively utilize the creative tooling. The editor's current reliance on purely accidental discoverability—requiring users to blindly guess that dragging the left mouse button orbits the camera—fails the accessibility standard required for a community-facing utility. While the controls themselves are functional, their affordances are non-existent.  
The application must first standardize its camera inputs to perfectly align with established industry defaults, actively preventing muscle-memory conflicts for users migrating from other 3D software ecosystems. The navigation model must enforce Alt \+ Left Mouse Button Drag or Middle Mouse Button Drag for camera orbit, Shift \+ Middle Mouse Button Drag or Right Mouse Button Drag for planar panning, and continuous Mouse Wheel Scroll for zoom interpolation. Historical terrain editors in the Spring engine ecosystem, such as SpringMapEdit, relied on incredibly dense, highly specific modifier key configurations stored entirely within external keys.cfg text files.15 This reliance on undocumented keystrokes (e.g., SHIFTCTRLALT114) alienated casual mappers. The new BAR Map Editor must categorically reject hidden bindings, ensuring all navigation is visually supported by the UI.  
To guarantee these controls are explicitly discoverable without external documentation, two specific interface elements must be introduced layered over the Central Viewport.  
The first element is a Viewport Navigation Gizmo positioned persistently in the top-right corner of the 3D viewport space. This interactive 3D axis gizmo, directly adapted from the navigation paradigms of Blender 1, features a spherical arrangement of color-coded coordinate nodes representing the X, Y, and Z axes. The gizmo rotates continuously in real-time to reflect the current physical camera angle relative to the world origin. Furthermore, it operates as an active input surface; clicking and dragging anywhere within the gizmo's invisible bounding sphere orbits the primary camera. Clicking a specific axis node instantly snaps the camera to a top-down or side orthographic view, a feature critical for the precise two-dimensional placement required for start positions and symmetric balancing.  
The second element is a transient First-Launch Cheat Sheet. Upon initializing the editor application for the very first time, a translucent egui::Window must overlay the bottom-left quadrant of the viewport. This window displays a concise, highly readable schematic of the primary camera bindings. A prominent interaction button allows the user to dismiss the window, subsequently storing a persistent flag in the local application configuration state to prevent its reappearance on subsequent software launches. Additionally, pressing the standard ? key at any moment during operation must overlay a comprehensive keyboard shortcut map across the entire screen, ensuring reference material is always instantly available.

## **7\. Build / preview workflow**

The terminal objective of the BAR Map Editor is to synthesize and output a fully playable .sd7 archive file compatible with the Recoil engine. The compilation and packaging process represents the culmination of the designer's effort, yet the current layout relegates this crucial execution block to the absolute bottom of a vertically scrolling list, rendering it functionally invisible to new users.  
Modern integrated development environments and commercial game engines uniformly position compilation, execution, and contextual preview functionalities at the absolute top of the interface hierarchy. The BAR Map Editor must adopt this layout convention to intuitively guide the user toward the completion of the mapping loop.  
The right-hand sector of the newly established Top Bar must be exclusively dedicated to the map lifecycle execution strip. This strip houses three distinct interactive components operating sequentially.

| Execution Component | Interface Behavior | Output Consequence |
| :---- | :---- | :---- |
| **Validation Badge** | Evaluates structural integrity of map data prior to build. Displays green checkmark, yellow warning, or red error icon. | Prevents packaging of broken archives. Hovering exposes textual errors (e.g., "Warning: Map dimensions exceed standard limits"). |
| **Build & Install** | Primary egui::Button. Text transitions to "Building..." with loading spinner during execution. | Compresses directory structure, evaluates heightmap formats, and outputs final .sd7 archive to the system. |
| **Preview in BAR** | Contextual execution command. Active only after a successful build loop completes. | Launches the Recoil engine executable as a child process, instantly loading the compiled archive for rapid iteration testing. |

By elevating these components to the top level, the interface implicitly instructs the user that building the map is a continuous, iterative process rather than a final, hidden step. The validation badge serves as an active safeguard, analyzing the map state (such as ensuring start positions exist) and intercepting fatal engine errors before the user wastes time launching a broken map into the game client.

## **8\. Procgen UX (the f(x, z) panel)**

The current procedural generation panel relies exclusively on a raw mathematical input field. This UX design generates severe workflow friction due to inevitable syntax typing errors and a complete lack of visual preview capability. The user is forced into a highly destructive loop: blindly typing an equation, applying it directly to the terrain, observing the catastrophic topological failure, and executing a heightmap undo command. This workflow must be aggressively transformed into a safe, non-destructive, and visually responsive procedural ecosystem.  
The underlying software architecture requires the implementation of real-time background validation for the mathematical expression. As the user injects keystrokes into the egui::TextEdit field, a parallel background thread must continuously attempt to parse the Abstract Syntax Tree (AST) of the mathematical formula. If the parser encounters an invalid token or missing operator, the text field must immediately respond with visual feedback. The egui styling should draw a severe red underline beneath the specific syntax causing the parsing fault. Simultaneously, a localized error text block positioned directly below the input field must output the exact human-readable parser error string. Most importantly, the primary "Apply" execution button must be hard-disabled (add(egui::Button::new("Apply").interactive(false))) for the duration of the invalid syntax state, absolutely preventing application crashes caused by passing malformed instructions into the procedural generator.  
Users must never be expected to apply a mathematical formula to the primary geometric canvas blindly. The procedural generation panel, now relocated into the Right Inspector, must incorporate a dedicated 256x256 pixel egui::Image acting as a real-time non-destructive thumbnail preview. As a valid formula is synthesized, a low-resolution processing pass of the mathematics is executed against a normalized coordinate grid. The resulting topological heightmap is rendered as a standard greyscale image map within this localized thumbnail. This paradigm provides instant, continuous feedback on the macro topology of the equation without triggering massive, computationally expensive updates to the 3D viewport's heavy wgpu vertex buffers. Only when the thumbnail accurately reflects the designer's mathematical intention does the user click "Apply" to commit the transformation to the primary operational canvas.  
Recognizing that the target demographic is a community of real-time strategy map designers rather than dedicated technical artists or shader programmers, the freeform mathematical input must be visually deprioritized. Analyzing the success of Blender's procedural terrain generation plugins (such as A.N.T. Landscape), the interface relies heavily on presets to abstract underlying noise algorithms. The top of the procedural generation panel must be dominated by a prominent Dropdown menu labeled Generation Presets. This menu should contain a curated library of pre-written equations representing standard terrain topologies, such as Crater, Rolling Hills, Central Mesa, and Simplex Noise Base. Selecting a preset from the menu automatically populates the underlying text field with the relevant mathematical syntax, allowing the user to safely tweak isolated numerical values rather than forcing them to draft complex trigonometric relationships from scratch.

## **9\. The first 30 seconds**

The initial user experience during the first thirty seconds of interaction definitively dictates software retention rates. Currently, the editor presents a standard modal configuration wizard and subsequently deposits the user into a potentially empty, flat, or confusing graphical state. The interactive journey from application launch to mechanical understanding must be highly curated to prevent user abandonment.  
When the application initializes, presenting the "New Project" modal wizard remains appropriate, but the default configuration values contained within must be highly opinionated. The wizard should default to a standard 16x16 map dimension, a universally recognizable biome preset, and a pre-configured 8v8 point-symmetry structural layout.  
Upon clicking the confirmation execution, the user must not be confronted with a flat, featureless grey plane. The editor must automatically execute a mild procedural simplex noise function in the background to generate immediate, rolling terrain topology. Furthermore, it must automatically instantiate sixteen start position markers across the generated symmetry axes in valid spatial locations. This establishes a "Shippable Demo State." Within exactly five seconds of closing the initial wizard, the user is presented with a fully playable, three-dimensional competitive map environment.  
With the demo map generated, the graphical interface must actively draw attention to its operational paradigm. A transient egui::Window overlay appears in the geometric center of the viewport, containing three simple, targeted directives establishing the core application loop:

1. "Sculpt the terrain profile," accompanied by a visual arrow pointing to the Sculpt icon located within the new left-hand Global Toolbar.  
2. "Verify spatial symmetry," accompanied by a visual arrow targeting the Symmetry configuration block centered in the Top Bar.  
3. "Compile and test your map," accompanied by an arrow highlighting the "Build & Install" button located in the top-right execution strip.

By providing the user with a map environment that is already structurally valid and ready for engine compilation, the interface implicitly assures the user that they cannot fatally break the system. They are psychologically freed to select the Sculpt tool, deform a localized geographical feature, and immediately trigger the Build execution to observe their custom geometric deformation running natively inside the Recoil engine. This curated onboarding pipeline completely eliminates the "blank canvas paralysis" that heavily afflicts standard open-ended DCC tools, accelerating the user's time-to-first-map metric from potential hours down to less than a single minute.
